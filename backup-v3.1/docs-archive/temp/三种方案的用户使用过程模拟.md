## 方案一用户使用过程

### 第一天：第一次使用

#### 场景：下载和安装

**下午3点，你刚刚了解到这个桌面宠物项目...**

你从GitHub下载了源代码压缩包，解压到电脑上。双击打开文件夹，看到了这样的结构：

```
claude_pet/
├── monitor_claude.py
├── desktop_pet.py
└── gifs/
    ├── idle.gif
    ├── working.gif
    ├── thinking.gif
    └── error.gif
```

**你的内心想法**：
> "文件夹这么简单？看起来好像不太复杂..."

你按照README的说明，打开了命令行（Windows上叫"命令提示符"，Mac上叫"终端"），输入了：

```bash
pip install pillow psutil
```

等待了大约1分钟，安装完成，没有报错。

#### 场景：第一次启动

你双击运行了 `desktop_pet.py`，会发生什么？

**你看到的画面**：

```
┌─────────────────────────────────────────────┐
│                                             │
│                                             │
│           ┌───────────────┐                 │
│           │    🐱         │  ← 一个小猫图片  │
│           │   (待机动画)   │                 │
│           └───────────────┘                 │
│                                             │
│         当前状态：等待连接...                │
│                                             │
│                                             │
└─────────────────────────────────────────────┘
```

一个小窗口出现在屏幕右下角，里面有一只可爱的小猫正在做待机动作。

窗口可以拖拽到任意位置，右键点击会弹出菜单，可以选择"退出"。

#### 场景：观察使用过程

你打开了Claude Code，开始正常工作...

**5分钟后...**

你注意到小猫的状态变了：

```
┌─────────────────────────────────────────────┐
│                                             │
│           ┌───────────────┐                 │
│           │    🐱         │                 │
│           │  (敲键盘动画)  │  ← 动作变了！  │
│           └───────────────┘                 │
│                                             │
│         当前状态：工作中...                   │
│                                             │
└─────────────────────────────────────────────┘
```

**你的内心想法**：
> "哇，它真的在动！看来Claude Code正在工作。"

又过了10分钟，Claude Code执行完任务，进入等待状态...

```
┌─────────────────────────────────────────────┐
│                                             │
│           ┌───────────────┐                 │
│           │    🐱         │                 │
│           │  (发呆动画)    │  ← 待机状态    │
│           └───────────────┘                 │
│                                             │
│         当前状态：等待中...                  │
│                                             │
└─────────────────────────────────────────────┘
```

### 第二天：发现问题

**场景：局限性体验**

你发现了一个问题...

你同时打开了Claude Code和一个普通记事本文件在写日记。Claude Code在认真工作，记事本只是偶尔敲几个字。

但是桌面宠物显示的状态是"工作中..."，并没有区分是哪个程序在工作。

**你内心可能的os**：
> "嗯...它好像不能区分具体是Claude在工作，还是其他程序在工作。只要CPU有动静，它就认为在工作。"

不过，你回想了一下：
> "其实也还好，反正我主要就是想看Claude是不是在工作。CPU有动静大概率就是在工作嘛。"

### 第一周使用总结

| 体验点 | 评分 | 你的感受 |
|--------|------|----------|
| 安装难度 | ⭐⭐⭐⭐⭐ | 太简单了，5分钟搞定 |
| 使用难度 | ⭐⭐⭐⭐⭐ | 打开就能用，不用任何设置 |
| 功能体验 | ⭐⭐⭐ | 基本够用，但不够精细 |
| 稳定性 | ⭐⭐⭐⭐⭐ | 一周没出过问题 |
| 整体满意 | ⭐⭐⭐⭐ | 性价比很高 |

### 第二天：发现问题

**场景：跨平台问题（如果你用Mac）**

你把项目分享给使用Mac的朋友...

**朋友的反馈**：
> "启动后弹出一个窗口选择器，但窗口列表是空的！我明明开着Claude Code啊！"

你查了一下发现...

**问题原因**：
- pywinauto 是专门为Windows设计的库
- 在Mac上，它无法很好地读取终端窗口的内容
- 窗口列表可能显示不完整

**解决方案**：
- 方案1：让Mac用户使用"手动输入窗口标题"功能
- 方案2：切换到方案一（放弃精准监控）
- 方案3：换用其他技术实现

**你的内心想法**：
> "原来方案二只能在Windows上完美运行...亏我还想推荐给Mac用户。"

### 第三天：遇到新问题

**场景：Windows更新后的问题**

你的Windows系统自动更新了...

再次打开桌面宠物，弹出了一个错误提示：

```
┌─────────────────────────────────────────────────┐
│                    ⚠️ 错误                       │
│                                                 │
│      无法连接到窗口 "Terminal - Claude Code"     │
│                                                 │
│      可能的原因：                                │
│      • 窗口已关闭                                │
│      • 窗口标题已更改                            │
│      • 系统权限不足                              │
│                                                 │
│         [ 重新选择窗口 ]  [ 退出 ]               │
└─────────────────────────────────────────────────┘
```

**发生了什么？**

Windows更新后，某些API的调用方式变了，导致pygetwindow和pywinauto有时候"认不出"窗口了。

你点击"重新选择窗口"，重新绑定了一下，问题解决了。但内心有点不爽...

**你的内心想法**：
> " Windows每次更新都可能出问题...这个项目以后维护起来可能有点麻烦。"

### 第一周使用总结

| 体验点 | 评分 | 你的感受 |
|--------|------|----------|
| 安装难度 | ⭐⭐⭐ | PyQt5安装时间有点长 |
| 初始配置 | ⭐⭐⭐⭐ | 窗口选择功能很直观 |
| 功能体验 | ⭐⭐⭐⭐⭐ | 太棒了，能精确知道Claude在做什么 |

| 整体满意 | ⭐⭐⭐⭐ | 功能很棒，但有点不稳定 |
