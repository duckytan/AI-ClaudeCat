## ⚠️ 发现的问题与改进建议

### 问题清单

| 序号 | 问题类型 | 具体表现 | 影响 |
|-----|---------|---------|------|
| 1 | 返工频繁 | 多次"回退"、"重做" | 效率损失 |
| 2 | 情绪化表达 | "你有没有病啊"等批评 | 可能影响AI响应质量 |
| 3 | 需求确认不足 | "先看看效果"而非先定义标准 | 验收标准模糊 |
| 4 | 口头回退 | "回退到之前的方案" | 不易追溯版本 |
| 5 | 重复批评 | 同一错误多次批评 | 未能建立防错机制 |

## ✅ 改进建议

### 1. 建立「需求确认」机制

**问题**：AI理解与预期不符，导致返工

**建议**：
```
每次需求 → AI复述 → 你确认 → AI才动手
```

**示例指令**：
```
"在我确认理解正确之前，不要修改任何代码"
```

---

### 2. 用「版本快照」替代口头回退

**问题**："回退"指向不明，难以追溯

**建议**：
```bash
# 每次大改前
git commit -am "UI优化-v1（保存点）"

# 不满意时回退
git reset --hard HEAD@{1}
```

**配置文件** `.git-message-template`：
```
# 项目协作规则
1. 任何修改前，必须先读取相关代码文件
2. 涉及部署，先查 Vercel/平台官方文档
3. 重大修改前创建 git commit 快照
4. 禁止直接运行未经检查的代码
5. 「能用AI处理 → 优先AI，脚本仅传参」
```

---

### 3. 定义「可衡量的验收标准」

**问题**："先看看效果"、"没任何变化"过于模糊

**建议模板**：
```
【任务目标】：[一句话]
【期望结果】：[具体、可衡量]
【禁止事项】：[1、2、3]
【强制要求】：[1、2、3]
【确认机制】：先给我看代码，我确认后再运行
```

**示例**：
```
【任务目标】：修改卡片渐变背景
【期望结果】：
  1. 渐变背景覆盖整个卡片区域
  2. 文字清晰可见，不被背景遮挡
  3. 不影响已有功能
【禁止事项】：不修改其他页面、不改变布局结构
【强制要求】：1. 用 Context7 查 Tailwind 渐变语法
              2. 创建 git 快照再修改
【确认机制】：先给我看 CSS 代码，我确认后再运行
```

---

### 4. 建立「错误清单」防止再犯

**问题**：同一错误反复出现

**建议**：
```
错误记录 → 原因分析 → 解决方案 → 更新规范/检查清单
```

**错误清单模板**：
```markdown
## 错误记录

| 日期 | 错误 | 原因 | 解决方案 | 预防措施 |
|-----|------|-----|---------|---------|
| 2025-01 | Vercel runtime 写错 | 没查官方文档 | 改为 'edge' | 强制要求查 Context7 |
```

---

### 5. 减少情绪化表达

**问题**："你有病啊"等批评可能影响协作效率

**替代方案**：

| 现状 | 改为 |
|-----|------|
| "你有病啊" | 「理解有误，请重新阅读需求」 |
| "改了个寂寞" | 「结果与预期不符，请对比修改前后」 |
| "一堆错误" | 「有3个编译错误，请逐个修复」 |

**目的**：让AI更精准定位问题，而非感到被攻击

---

### 6. 区分「AI问题」与「需求问题」

| 情况 | 责任方 | 改进措施 |
|-----|-------|---------|
| AI 不读 Context7 | AI侧 | 强制要求 AI 先查文档 |
| 需求太抽象 | 你侧 | 用 1、2、3 分点写 |
| 同一错误反复犯 | 双方 | 建立「错误清单」 |
